## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

基于 plan.app.md 中的任务列表，逐步执行实施，直到 core 和当前 feature 的 E2E 测试用例通过。

## 执行步骤

### 1. 加载上下文

1. 读取当前分支对应的 `specs/{branch-name}/spec.app.md` 和 `specs/{branch-name}/plan.app.md`
2. 读取项目的 `{{MEMORY_PATH}}` 了解项目架构和约定
3. 解析 plan.app.md 中的任务列表

### 2. 任务执行

按照 plan.app.md 中定义的阶段和顺序执行任务：

#### 执行规则

1. **按阶段执行**: 完成当前阶段所有任务后再进入下一阶段
2. **遵循依赖**: 按照任务依赖关系顺序执行
3. **实时更新**: 完成一个任务后立即在 plan.app.md 中标记为 `[x]`
4. **错误处理**: 遇到问题时停止并报告，等待用户指示

#### 任务完成标准

每个任务完成需满足：
- 代码已编写并保存
- 无语法错误
- 符合项目代码风格（参考 {{MEMORY_PATH}}）

### 3. E2E 测试执行

#### 3.1 测试方法

请参考 README.md 开始 E2E 测试，如果没有相关 E2E 测试方案，请和用户确认并创建。

#### 3.2 测试结果处理

**如果测试通过**:
- 标记所有测试任务为完成
- 报告成功

**如果测试失败**:
1. 分析失败原因
2. 定位问题代码
3. 修复问题
4. 重新运行测试
5. 重复直到通过

### 4. 智能记忆更新（自我学习）

所有任务完成且测试通过后，执行智能记忆更新，形成自我学习闭环。

#### 4.1 记忆更新触发条件

当出现以下**任意**情况时，更新 `{{MEMORY_PATH}}`：

| 条件 | 示例 | 优先级 |
|------|------|--------|
| **重复犯错** | 开发过程中同一错误出现 2 次以上 | 高 |
| **非显而易见的解决方案** | 需要大量调试才找到的解决方案 | 高 |
| **架构决策** | 新的模式、约定或技术选型 | 中 |
| **陷阱与坑** | 发现的意外行为、边界情况 | 中 |
| **过时信息** | 现有记忆内容不再准确 | 高 |
| **新的最佳实践** | 实现过程中发现的更好方法 | 中 |

#### 4.2 记忆内容分类

将记忆更新组织为以下类别：

```markdown
## 项目架构
- 技术栈和版本
- 目录结构约定
- 使用的关键设计模式

## 开发约定
- 代码风格规则
- 命名约定
- 文件组织模式

## 常见陷阱（自动学习）
- [日期] 问题: {描述}
  - 症状: {出了什么问题}
  - 根因: {为什么会发生}
  - 解决方案: {如何修复/避免}

## API 与集成说明
- 外部服务的特殊行为
- 认证模式
- 数据格式要求

## 测试指南
- 测试文件位置
- 测试命令
- 常见测试失败及修复方法

## 性能考虑
- 已知瓶颈
- 使用的优化技术
```

#### 4.3 记忆更新原则

**应该更新记忆的情况：**
- 犯了一个花费大量调试时间的错误
- 发现与文档描述不符的行为
- 找到了应该复用的模式
- 现有记忆内容错误或过时
- 学到了未来开发需要的项目特定知识

**不应该更新记忆的情况：**
- 通用编程知识（如 "使用 async/await 处理 Promise"）
- 不太可能再次发生的一次性修复
- 没有团队共识的个人偏好
- 应该被正确修复的临时解决方案

#### 4.5 记忆更新检查清单

完成实施前，请验证：

- [ ] **回顾开发历程**: 犯了什么错误？什么花费的时间比预期长？
- [ ] **检查重复模式**: 类似的问题是否多次出现？
- [ ] **验证现有记忆**: 当前内容是否有不再正确的？
- [ ] **记录非显而易见的解决方案**: 其他开发者是否会在同样的问题上挣扎？
- [ ] **记录架构决策**: 有什么新的模式应该成为标准？

### 5. 进度报告

每完成一个任务阶段，报告：
- 已完成任务数 / 总任务数
- 当前阶段状态
- 遇到的问题（如有）

## 实施原则

### 代码质量

- 遵循 readme.md 中定义的代码风格

### 错误处理

- 遇到不明确的问题时询问用户
- 不要假设或跳过问题
- 保留错误日志便于调试

### 测试驱动

- E2E 测试是最终验收标准
- 测试失败时优先分析测试代码是否正确
- 确保测试覆盖 spec.app.md 中定义的 Golden Case
- **必须实现所有测试用例**: 不要因为成本或时间原因跳过 spec 中设计的测试用例，这些用例是检验功能是否正常的关键标准

## 完成标准

实施完成需满足以下所有条件：

1. **所有任务完成**: plan.app.md 中所有任务标记为 `[x]`
2. **E2E 测试通过**: spec.app.md 中定义的所有测试用例通过
3. **记忆已更新**: 自我学习更新已应用到 `{{MEMORY_PATH}}`

> **⚠️ 重要**: 任务完成后请返回检查 plan.app.md 中的所有 task 是否全部完成（标记为 `[x]`）。如果有未完成的任务，必须继续执行直到全部完成！

## 输出

完成后报告：
1. 任务完成情况统计
2. E2E 测试结果
3. 修改的文件列表
4. **记忆更新内容**（学到了什么并记录了什么）

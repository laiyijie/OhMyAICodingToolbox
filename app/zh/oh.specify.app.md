description: 创建功能规范，包含用户场景、E2E 测试用例和澄清问题
---

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

基于用户提供的功能描述，创建一个简洁的功能规范文档。

## 执行步骤

### 1. 生成分支名称

分析功能描述，生成 2-4 个词的简短名称：
- 使用 kebab-case 格式（如 `add-user-auth`、`fix-payment-bug`）
- 保留技术术语和缩写

### 2. 创建分支和规范文件

```bash
# 获取最新远程分支
git fetch --all --prune

# 查找同名分支的最高编号
# 检查远程分支、本地分支、specs/ 目录

# 创建新分支（编号+1）
git checkout -b {number}-{short-name}

# 创建规范目录和文件
mkdir -p specs/{number}-{short-name}
```

### 3. 编写规范文件

在 `specs/{number}-{short-name}/spec.app.md` 中编写以下内容：

```markdown
# {功能名称}

## 1. 用户场景

描述用户如何使用此功能的完整流程。

### 场景 1: {主要场景名称}

**作为** {用户角色}
**我想要** {执行的操作}
**以便** {期望的结果}

**流程**:
1. 用户 {操作步骤 1}
2. 系统 {响应步骤 1}
3. ...

### 场景 2: {次要场景名称}（如有）

...

## 2. E2E 测试用例（Golden Case）

仅包含核心 Happy Path 测试用例，不包含错误处理和边界情况。

### TC1: {测试用例名称}

**前置条件**:
- {条件 1}
- {条件 2}

**测试步骤**:
1. {用户操作 1}
2. {用户操作 2}
3. {用户操作 3}

**预期结果**:
- {期望的系统响应 1}
- {期望的系统响应 2}

### TC2: {测试用例名称}（如有）

**前置条件**:
- ...

**测试步骤**:
1. ...

**预期结果**:
- ...

## 3. 待澄清问题

> **注意**: 最多 3 个问题，仅针对无法合理推测的关键决策。

### Q1: {问题标题}

**上下文**: {为什么需要澄清}

**选项**:
| 选项 | 描述 | 影响 |
|------|------|------|
| A | {选项 A} | {影响} |
| B | {选项 B} | {影响} |

### Q2: {问题标题}（如有）

...
```

## 规范编写原则

### 用户场景

- 专注于用户需要**什么**和**为什么**
- 避免实现细节（不涉及技术栈、API、代码结构）
- 覆盖主要使用流程

### E2E 测试用例

- 只包含 Golden Case（Happy Path）
- 不包含错误处理和边界情况
- 描述用户操作步骤和预期结果，不涉及代码
- 测试用例应该端到端，验证完整用户流程

### 待澄清问题

限制为最多 3 个问题，仅在以下情况提问：
- 显著影响功能范围
- 存在多个合理解释且影响不同
- 无法基于上下文做出合理默认选择

**不需要提问的情况**（使用合理默认值）：
- 标准的错误处理方式
- 行业标准的性能指标
- 常见的 UI 交互模式

## 输出

完成后报告：
1. 分支名称
2. 规范文件路径
3. 需要用户回答的澄清问题（如有）

**下一步**: 用户确认规范后，运行 `/oh.plan.app` 创建技术方案

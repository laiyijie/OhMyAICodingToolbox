## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

基于用户提供的被测功能/API 描述，创建一个 E2E 测试规范文档。此命令用于**专门管理 E2E 测试用例的项目**。

## 执行步骤

### 1. 探索项目（必须）

在创建规范之前，先了解项目现有情况：

1. **查找现有规范文档**：检查项目是否有现有的测试规范、需求文档或类似文件
2. **了解规范格式**：如果项目已有规范文档，分析其格式和结构
3. **查看测试目录**：了解项目的测试组织方式

### 2. 生成分支名称

分析测试目标，生成 2-4 个词的简短名称：
- 使用 kebab-case 格式（如 `test-user-login`、`test-payment-api`）
- 保留技术术语和缩写

### 3. 创建分支和规范文件

```bash
# 获取最新远程分支
git fetch --all --prune

# 查找同名分支的最高编号
# 检查远程分支、本地分支、specs/ 目录

# 创建新分支（编号+1）
git checkout -b {number}-{short-name}

# 创建规范目录和文件
mkdir -p specs/{number}-{short-name}
```

### 4. 编写测试规范文件

在 `specs/{number}-{short-name}/spec.e2e.md` 中编写规范。

#### 4.1 规范核心要素

无论采用什么格式，规范必须包含以下核心要素：

| 要素 | 说明 | 必须 |
|------|------|------|
| 被测系统描述 | 测试什么系统/功能，入口在哪里 | ✓ |
| 测试范围 | 包含什么、排除什么 | ✓ |
| 测试用例 | 具体的测试场景和预期结果 | ✓ |
| 测试数据 | 需要什么数据，如何准备 | 视情况 |
| 环境依赖 | 需要什么环境和服务 | 视情况 |
| 待澄清问题 | 需要用户确认的决策点 | 视情况 |

#### 4.2 规范格式选择

- **如果项目有现有规范格式**：遵循现有格式
- **如果项目没有现有格式**：根据测试复杂度选择合适的详细程度

#### 4.3 测试用例编写原则

- 按优先级排序（P0 > P1 > P2）
- 测试步骤应该具体、可执行
- 预期结果必须可验证

## 规范编写原则

### 被测系统描述

- 清晰描述被测试的目标系统
- 包含必要的访问信息和文档链接
- 明确测试入口

### 测试范围

- 明确界定测试边界，避免范围蔓延
- 列出排除项及其原因

### 测试用例

- 描述用户操作步骤和预期结果
- 不涉及具体实现细节（选择器、代码等）
- 包含必要的测试数据示例

### 待澄清问题

限制为最多 3 个问题，仅在以下情况提问：
- 显著影响测试范围
- 存在多个合理解释且影响不同
- 无法基于上下文做出合理默认选择

## 输出

完成后报告：
1. 分支名称
2. 规范文件路径
3. 测试用例数量和优先级分布
4. 需要用户回答的澄清问题（如有）

**下一步**: 用户确认规范后，运行 `/oh.plan.e2e` 创建测试方案

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 概述

基于 plan.e2e.md 中的任务列表，逐步执行 E2E 测试实施，直到所有测试用例通过。

## 执行步骤

### 1. 加载上下文（必须）

1. 读取当前分支对应的 `specs/{branch-name}/spec.e2e.md` 和 `specs/{branch-name}/plan.e2e.md`
2. 读取项目的 `{{MEMORY_PATH}}` 了解项目架构和约定
3. 解析 plan.e2e.md 中的任务列表

### 2. 任务执行

按照 plan.e2e.md 中定义的阶段和顺序执行任务：

#### 执行规则

1. **按阶段执行**: 完成当前阶段所有任务后再进入下一阶段
2. **遵循依赖**: 按照任务依赖关系顺序执行
3. **实时更新**: 完成一个任务后立即在 plan.e2e.md 中标记为 `[x]`
4. **参考现有代码**: 编写新代码时，参考 plan.e2e.md 中列出的现有模式
5. **错误处理**: 遇到问题时停止并报告，等待用户指示

#### 任务完成标准

每个任务完成需满足：
- 代码已编写并保存
- 无语法错误
- 与项目现有代码风格保持一致

### 3. 测试执行

#### 3.1 运行测试

根据项目的测试配置运行测试（参考 README 或 package.json 中的测试命令）。

#### 3.2 测试结果处理

**如果测试通过**:
- 标记所有测试任务为完成
- 继续执行回归测试

**如果测试失败**:
1. 分析失败原因（查看错误信息和堆栈）
2. 定位问题代码
3. 修复问题
4. 重新运行测试
5. 重复直到通过

#### 3.3 回归测试（必须）

运行项目的完整测试套件，确保新代码不破坏现有功能。

### 4. 智能记忆更新（必须）

所有任务完成且测试通过后，执行智能记忆更新。

#### 4.1 记忆更新触发条件

当出现以下**任意**情况时，更新 `{{MEMORY_PATH}}`：

| 条件 | 优先级 |
|------|--------|
| 同一错误出现 2 次以上 | 高 |
| 解决方案需要大量调试时间 | 高 |
| 现有记忆内容不再准确 | 高 |
| 发现框架/工具的意外行为 | 中 |
| 找到有效的模式或策略 | 中 |

#### 4.2 应该记录的内容

- 花费大量调试时间的问题及解决方案
- 与文档描述不符的行为
- 有效的选择器策略或等待模式
- 项目特定的测试约定
- 可复用的模式

#### 4.3 不应该记录的内容

- 通用测试知识
- 框架官方文档已详细说明的内容
- 一次性的临时修复
- 特定测试用例的细节

### 5. 进度报告

每完成一个任务阶段，报告：
- 已完成任务数 / 总任务数
- 当前阶段状态
- 遇到的问题（如有）

## 实施原则

### 代码质量

- 与项目现有代码风格保持一致
- 参考 plan.e2e.md 中列出的现有测试模式
- 复用现有的基础类和工具函数

### 测试隔离

- 每个测试用例应该独立运行
- 不依赖其他测试的执行顺序
- 按照 plan.e2e.md 中的数据策略管理测试数据

### 问题处理

- 遇到不明确的问题时询问用户
- 不要假设或跳过问题
- 保留错误日志便于调试

## 完成标准

实施完成需满足以下所有条件：

1. **所有任务完成**: plan.e2e.md 中所有任务标记为 `[x]`
2. **新增测试通过**: spec.e2e.md 中定义的所有测试用例通过
3. **回归测试通过**: 现有测试套件全部通过
4. **记忆已更新**: 自我学习更新已应用到 `{{MEMORY_PATH}}`

> **⚠️ 重要**: 任务完成后请返回检查 plan.e2e.md 中的所有 task 是否全部完成（标记为 `[x]`）。如果有未完成的任务，必须继续执行直到全部完成！

## 输出

完成后报告：
1. 任务完成情况统计
2. 测试执行结果
3. 修改/创建的文件列表
4. **记忆更新内容**（学到了什么并记录了什么）
